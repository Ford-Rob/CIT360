Typically, when doing an Object-Oriented Analysis & Design, you don't require coding. However, the best
way to truly understand the concepts is to write code.

The source starting code is found in Turtle5.zip file and contains an Adobe Acrobat file and the static class
diagram. The base code has some problems from the perspective of encapsulation, cohesion and coupling.
If you want to work on moving code to an OO and reuse pattern, then you should modify the code to
guarantee the following:

  1. Effective encapsulation is consistent.
  2. Identification and localization by package the reusable code components.
  3. Effective cohesion of classes.
  4. Localization of coupling withing packages.

Some types of problems and fixes you may need to look for:

Problem Potential Solution Rationale
Dependencies on class static constant
variables are examples of coupling
when they are not arguments passed
into a method of the class owning the
static variables. 		Classes that publish static constants
typically provide public "getters" and
"setters" methods for those variables
that may be required to effectively leverage
objects that are internally coupled within a
package. 		Publishing class static constant
variables is done ONLY when they
are required to guarantee
structured dependencies in the
planned behavior of the class
methods.
 
Published class static constant
variables should only alter the
runtime behavior of the object.
If two or more methods are required
to effect published events, then
the called object is not encapsulated
or cohesive. 		If an object has two methods that effect and
action event, then the two methods should
be demoted in scope to private access and
a wrapper method provided that internally
calls the two private methods in the correct
temporary order.
 Public and protected object methods
should manage events. Events are
published behaviors. Events may be
managed by one or many private
methods.
